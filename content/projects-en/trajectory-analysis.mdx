## Trajectory Analysis Module Design & Productization

### Project Overview

<img src="/images/flow_animation.gif" alt="Trajectory Analysis Module demo" style={{ width: "100%", borderRadius: "8px" }} />

Existing heatmaps could show where visitors lingered but couldn't explain in what order they moved or what flow patterns they exhibited.
I designed a FlowMap prototype that reconstructs coordinate logs into sequence units and classifies similar trajectories, generalized per-client store structure differences via YAML config, then productized into heatmap / journeymap / traffic_map modules.
Extended trajectory data from density-focused analysis to sequence/type/flow analysis, building 3 production modules and a reusable analysis pipeline applicable to actual client data.

### Problem / Root Cause / Outcome

| Problem | Root Cause | Outcome |
|---------|-----------|---------|
| **Heatmap-only trajectory interpretation limits** | Existing visualization showed dwell density but couldn't explain movement order or similar path patterns | Reconstructed coordinate logs as sequences, extending to **journeymap / traffic_map** structure enabling order and flow analysis |
| **Inefficient per-client store adaptation** | Store layouts and zone definitions differed by client; code-modification-centric approach had low scalability and reusability | Separated zones, coordinate ranges, and visualization criteria into **YAML config** for settings-based generalization |
| **Trajectory clustering vulnerable to outliers** | Short stays, immediate exits, and prolonged single-zone stays distorted central trajectories, reducing cluster representativeness | Adopted **KMedoids** instead of KMeans for outlier-robust trajectory type classification |
| **Poor communicability of coordinate-based results** | Results remaining as coordinate sequences were hard for non-experts to read and difficult to use in client communication | Added **Sankey diagram** translating inter-zone movement into intuitive flow visualization |

## Approach

### 1) Reinterpreted Coordinate Logs as Sequences, Not Aggregations

Heatmaps show frequently visited locations well but cannot explain in what order visitors experienced the space.
- Viewed coordinate logs not as simple point sets but reconstructed them as time-ordered trajectories.
- Then determined a method for comparing path similarity was needed, reviewing DTW, LCSS, NMF, and others.
- Explored sequence analysis structures that preserve actual movement order while enabling comparison.

Trajectory data was extended beyond dwell density analysis to behavioral data interpretable for visit order and movement patterns.

### 2) Generalized Per-Client Spatial Differences via Config

Clients like Shinsegae and Hyundai Innocean had different zone compositions and coordinate structures, making code-modification-based approaches poorly scalable.
- Separated per-store zone definitions, coordinate ranges, visualization criteria, and label information into YAML config rather than code.
- This allowed maintaining common analysis logic while adapting to per-client differences through settings changes alone.

The trajectory analysis module was organized not as client-specific code but as a reusable pipeline applicable to multiple spatial structures.

### 3) Differentiated Prototype into 3 Question-Focused Modules

The initial single FlowMap tried to answer all trajectory questions, making result interpretation and usage methods easy to conflate.
- Divided functionality based on the questions clients actually ask.
- `heatmap` answers "where do they stay," `journeymap` answers "in what order do they move," and `traffic_map` answers "where do they go from where."
Instead of maintaining a single experimental tool, I created an analysis module system repeatedly usable aligned to client questions.
