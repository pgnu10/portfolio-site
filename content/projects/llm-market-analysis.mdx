## LLM 기반 상권 분석 리포트 자동화

### 프로젝트 요약

<img src="/images/LLM Total_result.png" alt="상권 분석 리포트 결과" style={{ width: "100%", borderRadius: "8px" }} />

고객사는 행동 데이터 자체에는 만족했지만 대시보드를 직접 해석하고 외부 상권 정보까지 결합해 실행안을 도출할 여력이 없었다.  
이 문제를 해결하기 위해 기존의 수작업 상권 분석 리포트 제작 과정을 단계별로 구조화하고, 이를 **멀티 LLM Agent Flow와 RAG 기반 자동 리포트 시스템**으로 전환했다.  
결과적으로 사용자가 질문을 입력하면 외부 조사, 내부 데이터 분석, 전략 제안까지 포함한 리포트를 **10~15분 내 생성**할 수 있는 기능을 프로덕션 수준으로 구현했고, 고객사의 **계약 연장 및 추가 도입 논의**에도 활용됐다.

### 한눈에 보는 핵심 성과

* 수작업으로 3주 소요되던 상권 분석 리포트 제작 과정을 자동화

## 접근 방식

### 1) 수작업 분석을 먼저 표준화

<img src="/images/LLM preresearch.png" alt="상권 분석 사전 조사" style={{ width: "100%", borderRadius: "8px" }} />

처음부터 자동화된 상권 분석을 만들려고 하지 않고, 실제 고객용 상권 분석 리포트를 직접 작성하면서 분석 순서와 결과물 구조를 정리했다.
- 데이터 분석은 단순 요약이 아니라 순서가 있는 해석 과정이며,
- 분석의 모든 단계를 자동화할 수 있는 것은 아니기 때문이다.
이 과정을 통해 **무엇을 자동화할 수 있고 무엇은 사람이 판단해야 하는지** 경계를 먼저 확보했다.

### 2) 멀티 LLM 구조 채택

<img src="/images/LLM agent_flow.svg" alt="상권 분석 Agent Flow" style={{ width: "100%", borderRadius: "8px" }} />

초기에는 LLM에게 데이터 결합, 계산, 해석, 문장 생성까지 한 번에 맡겼지만 품질 저하와 수치 불일치가 반복됐다.
- 그래서 정합성이 중요한 수치 계산과 집계는 Python 코드로 분리하고, 
- LLM은 해석과 서술에 집중하도록 구조를 변경했다.
- 또한 멀티 LLM 구조를 채택해 외부 조사, 내부 데이터 분석, 전략 제안 역할을 분리했다.
이 과정을 통해 **분석가의 사고 과정을 시스템이 처리할 수 있는 단위로 분해**하여 설계했다

### 3) 품질 가드레일 설계

초기 파이프라인은 얼핏 보면 꽤 자연스러운 리포트를 만들었지만, 반복 실행해보면 수치와 결론이 조금씩 흔들렸다.
- 그래서 계산과 서술을 같은 단계에서 처리하지 않도록 했다.
- validator를 추가해 최종 수치가 원본 집계 결과와 일치하는지 검증하고
- JSON schema와 후처리를 적용해 스키마를 벗어난 경우 해당 단계로 되돌려 재생성하도록 설계했다.
결론적으로 **LLM은 해석과 서술**에 집중시키고, *정합성과 재현성이 필요한 부분은 코드로 통제*했다.

## 성과: 데이터 조회 서비스를 외부 맥락까지 결합한 해석형 분석 서비스로 전환
- 수작업 기준 3주 걸리던 상권 분석 리포트 제작 구조를 자동화하였다.
- 사용자의 질문 입력부터 완성 리포트 생성까지 15분 내 처리할 수 있는 프로덕션 수준의 기능을 설계했다.
- 멀티 LLM 역할을 분리 및 적용하였으며, 12개 Stage 프롬프트와 8단계 DAG 기반 Agent Flow 설계하였다.
- 동적 정보 처리를 위한 RAG 파이프라인 구성하였다.
- 아모레퍼시픽, 한섬 등 다수 고객사에 기능 제공하였으며, 고객 미팅에서 시연 후 계약 연장 및 추가 도입 논의에 활용하였다.

### 사용 기술과 역량
- **사용 기술**: `Python`, `JSON Schema`, `Streamlit`, `YAML/JSON`, `ThreadPoolExecutor`, `RAG`, `OpenAI o3-mini`, `GPT-4.1`, `Perplexity sonar-pro` 
- 고객 문제를 분석 서비스 구조로 재정의하는 능력
- 수작업 분석 프로세스를 자동화 가능한 단계로 분해하는 능력
- LLM을 만능 도구로 쓰지 않고 코드와 역할을 분리하는 설계 역량
- 정합성, 재현성, 설명 가능성을 고려한 품질 통제 역량
- 데이터 분석, AI 활용, 제품화 관점을 연결하는 실행력

## 1. 프로젝트 개요

상권 분석 리포트 자동화 프로젝트의 출발점은 고객사가 우리의 데이터를 잘 활용하고 있지 않다는 분석 결과였다.  
당시 최고가 플랜을 이용 중이었던 아모레퍼시픽은 그들이 지불하는 비용에 비해 매우 저조한 대시보드 서비스 이용률을 보였다. 이들의 계약 종료 시점이 다가옴에 따라 회사 내부적으로 고객사의 리텐션을 우려하는 목소리가 나오기 시작했고, 자연스레 고객사의 불만 사항을 선제적으로 파악하고 조치를 취해야 한다는 결론이 나왔다. 이를 위해 진행한 고객사와의 미팅에서 확인한 이유는 다음과 같았다.  
고객사의 불만은 우리의 데이터 품질에 대한 것이 아니라, **현업이 데이터를 직접 해석하고 활용할 시간이 없기** 때문에 데이터를 활용하지 못한다는 점이었다.

실제로 고객사의 실무진은 주에 한번씩 데이터를 보고 있었지만, 그 데이터를 실제 의사결정에는 쓰지 못하고 있었다. 대시보드에 쌓여있는 방문객 수, 체류시간, 유입 추이 같은 정보를 분석하고 해석하여 자신의 업무에 활용할 리소스가 부족했던 것이다. 실무진이 진정으로 궁금한 것은** 숫자 자체가 아니라, 그 숫자의 원인이 무엇인지, 그리고 그 문제를 해결하기 위해 무엇을 해야 하는지**였다.

이를 위해서는 단순한 행동 데이터만으로는 충분하지 않았다. 방문객이 줄었을 때, 이것의 원인이 매장 운영 문제인지 상권 자체의 침체인지 구분하려면 외부 상권 정보와 업종 맥락이 필수적이었다.  
그런데 이 과정을 사람이 직접 수행하면 한 지점을 조사하고 분석하는 데에 3주 이상이 걸렸고, 도저히 서비스로 확장 가능한 구조가 아니었다.

내가 맡은 역할은 이 수작업 분석 과정을 단계별로 구조화하고, **멀티 LLM Agent Flow + RAG + validator 기반 자동 리포트 시스템**으로 전환하는 것이었다.  
결과적으로 사용자가 질문을 입력하면 외부 조사, 내부 데이터 분석, 전략 제안까지 포함한 리포트를 즉각적으로 15분 내에 생성할 수 있는 프로덕션 수준의 기능을 설계했다.

## 2. 문제 배경

대시보드 활용도가 저조한 이유는 데이터 품질에 대한 불만이 아니라, **현업이 직접 해석할 시간이 없다는 것**이었다.

예를 들어 매장의 방문객 수가 감소했다는 데이터를 봤을 때, 현업 입장에서는 이런 질문이 따라온다.
* 우리 매장의 운영이나 상품 구성이 문제인 것인가?
* 상권 유동인구 자체가 줄어든 것인가?
* 경쟁 매장 변화가 있었던 것인가?
* 일시적 이벤트 영향인지 구조적 변화인지?

어떤 것이 원인이냐에 따라 해결 방안이 달라지는 것은 자명하다.  
하지만 기존 서비스는 내부 행동 데이터 중심이었기 때문에, 이 질문들에 답하려면 외부 상권에 대한 리서치가 추가로 필요했다.  
문제는 그 리서치 비용이 지나치게 크다는 점이었다.  
지역 상권 변화, 경쟁 매장, 업종 트렌드, 생활인구 데이터를 사람이 직접 수집하고 해석해 리포트까지 작성하면 사실상 맞춤형 컨설팅에 가까운 작업이 된다.  
일부 고객에게는 가능했지만, 다수의 고객사를 대상으로 한 SaaS 서비스로 반복하기에는 지속 가능하지 않았다.

결국 프로젝트의 핵심 과제는 분명했다.  
**외부 맥락을 포함한 해석형 분석 리포트를 반복 가능한 시스템으로 바꾸는 것.**

## 3. 내가 맡은 역할

이 프로젝트에서 내가 맡은 판단 영역은 크게 다섯 가지였다.

### 3-1. 문제 정의와 자동화 범위 설정

고객이 실제로 원하는 것이 단순 조회 기능이 아니라 해석형 분석 서비스라는 점을 정의했고, 이를 바탕으로 자동화의 목표를 정했다.  
핵심은 “리포트를 자동으로 만드는 것”이 아니라, **분석 프로세스를 제품 기능으로 바꾸는 것**이었다.

### 3-2. 수작업 분석 프로세스 구조화

초기 리포트를 직접 작성하면서 어떤 **데이터**가 필요하고, 어떤 **순서**로 해석해야 하며, 어떤 **논리**를 거쳐야 고객이 납득하는지를 정리했다.  
이후 자동화 파이프라인은 이 구조를 기반으로 설계됐다.

### 3-3. LLM 파이프라인 설계

12개 Stage **프롬프트**와 8단계 DAG 기반 **Agent Flow**를 설계했다.  
각 단계의 역할, 입력/출력 형식, 모델 선택 기준, 재생성 조건을 정의했다.

### 3-4. 품질 통제 설계

수치 일관성 문제를 해결하기 위해 계산과 서술을 분리하고, validator, JSON schema, Python 후처리를 도입했다.  
이 부분은 단순 프롬프트 튜닝보다 **시스템 신뢰성을 확보**하는 데 더 중요했다.

### 3-5. 리포트 UX 및 정보 구조 설계

최종 산출물이 고객에게 읽히고 활용될 수 있도록, 질문 입력부터 현황 리포트와 전략 리포트까지 이어지는 **정보 구조와 시각화 흐름**을 설계했다.

협업은 분명히 있었다.  
DAG 실행 엔진과 API 인프라, RAG 컨텍스트 구조 설계는 백엔드 엔지니어와 협업했고, UI 연동과 배포는 프론트엔드 엔지니어와 함께 진행했다.  

다만 내가 맡은 중심은 구현 세부보다 **분석 구조와 품질 기준을 설계하는 일**이었다.

## 4. 왜 수작업부터 시작했는가

돌이켜보면 이 프로젝트에서 가장 중요했던 결정은 "처음부터 곧장 자동화를 시도하지 않고, 수작업 분석 과정을 정리한 뒤 시작한 것"이었다.

수작업으로 리포트 하나를 만드는 데 데이터 정리, 외부 조사, 스토리라인 설계, 시각화, 문서 작성까지 포함해 2~3주가 들었다. 비효율적이었지만, 그 과정 덕분에 어떤 분석이 고객에게 실제로 의미가 있는지 알 수 있었으며, 오히려 자동화의 범위가 선명해졌다.  

이때 정리된 핵심은 두 가지였다.

첫째, **분석은 단순 요약이 아니라 순서가 있는 해석 과정**이라는 점이다.  
예를 들어 전체 방문객 구조를 먼저 보고, 이후 인구통계/공간/구역별 패턴으로 내려가며, 마지막에 외부 상권 맥락과 연결해야 고객이 이해하기 쉽다.

둘째, **모든 단계를 자동화할 수 있는 것은 아니라는 점**이다.  
수치 집계나 구조화된 비교는 코드와 프롬프트로 안정적으로 만들 수 있었지만, 고객 맥락에 맞는 표현이나 일부 전략 제안은 더 세심한 통제가 필요했다. 이 경계를 먼저 본 것이 이후 설계의 기반이 됐다.

즉, 자동화는 아이디어에서 출발한 것이 아니라 **실제 분석 노동을 해본 뒤 그 구조를 추출한 결과**였다.

## 5. 분석 프로세스를 어떻게 시스템으로 옮겼는가

### 5-1. 분석 프레임 정리

수작업 리포트에서 정리한 분석 논리를 바탕으로, 전체 흐름을 다음과 같이 구조화했다.

* 내부 행동 데이터 해석
* 외부 상권 및 업종 맥락 수집
* 현재 성과에 대한 원인 가설 도출
* 개선 방향과 전략 제안
* 리포트 형태로 전달

핵심은 “분석해줘” 같은 넓은 요청을 그대로 LLM에 넘기지 않는 것이었다. 대신 각 단계에서 무엇을 판단해야 하는지를 잘게 나눴다.  
예를 들어 변화율 계산, 상권 평균 비교, 이상 변화 탐지, 원인 가설 제시, 실행 제안은 각각 **요구하는 추론의 성격이 달랐다**. 이를 한 번에 시키면 결과는 그럴듯해 보여도 품질이 불안정했다.

### 5-2. 12 Stage 프롬프트 설계

그래서 프롬프트도 단계별로 분리했다.  
한 단계에서 하나의 역할만 수행하도록 설계했고, 중간 산출물은 JSON/YAML 형식으로 고정했다. 최종 단계에서만 문장형 리포트로 변환했다.

이 구조를 택한 이유는 다음과 같다.

* 단계별 실패를 분리할 수 있다.
* 어느 부분에서 품질이 흔들리는지 찾기 쉽다.
* 출력 형식을 강제하기 쉽다.
* 일부 단계만 재생성할 수 있어 운영성이 높다.

즉, 프롬프트를 많이 썼다는 사실보다 중요한 것은 **분석 사고 과정을 시스템이 처리할 수 있는 단위로 분해했다는 점**이었다.

## 6. 왜 멀티 LLM 구조를 택했는가

초기에는 단일 모델로 처리하는 방안도 검토했다. 구현이 쉽고 구조가 단순해 유지 보수가 쉽기 때문이다.  
그러나 실제로 돌려보면 각 단계가 요구하는 능력이 다르다는 점이 분명했다.

* 외부 상권 조사에는 최신화된 정보를 위한 실시간 검색이 중요했다.
* 데이터 기반 분석 단계에는 숫자와 조건을 바탕으로 원인과 해결 방안을 추론하는 능력이 필요했다.
* 최종 리포트 작성에는 문장 흐름과 보고서 톤을 안정적으로 만드는 능력이 중요했다.

이 때문에 각 역할별로 사용 모델을 나눴다.

* **외부 웹 리서치**: `sonar-pro`
* **데이터 해석과 분석 추론**: `o3-mini`
* **최종 리포트 생성과 문장 품질 확보**: `GPT-4.1`

여기서 중요한 것은 **분석 프로세스를 분해하여 최적의 역할을 수행하는 모델을 선택했다는 점**이다.  
그 결과 단일 모델보다 운영 구조는 복잡해졌지만 품질은 훨씬 더 안정적이 되었다.

## 7. 8단계 Agent Flow는 어떻게 설계했는가

최종 파이프라인은 DAG 기반 8단계 워크플로우로 구성했다.  
핵심은 **병렬화**와 **단계 분리**였다.

가장 먼저 외부 리서치 노드를 병렬로 실행했다.  
- 하나는 비즈니스 도메인과 브랜드 맥락을 조사했고, 
- 다른 하나는 해당 지역 상권의 유동인구, 경쟁 환경, 최근 이슈를 수집했다.  
이 단계는 수작업 시절 가장 시간이 오래 걸리던 부분을 크게 단축시켰다.

그다음 분석 계획 노드에서 전체 리포트의 목차와 질문 구조를 생성했다.  
가장 공을 들인 부분이 이 부분으로, 데이터 분석가가 데이터를 해석하고 분석하는 과정을 최대한 자세하게 반영하도록 설계했다.  
최종 리포트 결과의 품질에 가장 많은 영향을 준 부분이기도 하다.

<img src="/images/LLM analysis_node.png" alt="상권 분석 분석 계획 노드" style={{ width: "100%", borderRadius: "8px" }} />

이후 데이터 분석 노드가 내부 행동 데이터와 외부 상권 데이터를 해석했고, 성과 보고서와 전략 보고서가 순차적으로 생성되도록 구성했다.  
특히 전략 보고서 단계에서는 별도의 검색 쿼리를 생성해 **최신 트렌드와 벤치마크를 추가로 수집**하도록 설계했다.

이 구조의 장점은 다음과 같았다.

* 리서치와 분석을 병렬 처리해 시간을 줄일 수 있다.
* 각 단계의 입력과 출력을 명확히 관리할 수 있다.
* 일부 노드 실패 시 전체를 다시 돌리지 않아도 된.다
* 추후 특정 단계만 개선하거나 교체하기 쉽다.

## 8. 가장 큰 문제였던 수치 불일치를 어떻게 해결했는가

이 프로젝트에서 가장 위험했던 실패는 잘못된 문장이 아니라 **잘못된 숫자**였다.  
초기 파이프라인은 얼핏 보면 꽤 자연스러운 리포트를 만들었지만 반복 실행해보면 수치가 조금씩 흔들렸다. 어떤 경우에는 주중 평균 방문객을 계산하면서 일부 요일이 누락돼 실제보다 낮은 숫자가 보고서에 들어가기도 했다.  
문제는 LLM이 만들어낸 리포트의 문장은 대부분 자연스러웠기 때문에 잘못된 점을 찾아내기 어려워, 자칫하면 잘못된 정보를 제공할 위험이 있었다.

이 경험 이후 전반적인 구조를 재설계했다.

### 8-1. 계산과 서술 분리

가장 먼저 한 일은 **계산과 서술을 같은 단계에서 처리하지 않도록 한 것**이다.  
사칙연산이나 간단한 텍스트 가공 등의 프로그래매틱한 작업은 LLM을 이용하지 않고 코드로 수행하도록 했으며, LLM은 이미 확정된 결과를 해석하고 서술하는 역할만 맡도록 분리했다.  

이 과정에서 코드 구조가 복잡해지고 모듈간 Context 전환 비용이 증가하는 단점이 있었지만,  
LLM 운용 비용이 감소하고 실제 운영에서 훨씬 더 안정적인 결과물을 제공한다는 강력한 이점이 있었기 때문에 선택했다.

### 8-2. validator 도입

그다음 **validator**를 도입했다.  
최종 리포트에 쓰인 수치가 원본 집계 결과와 일치하는지 검증하고, 스키마를 벗어난 경우 해당 단계로 되돌려 재생성하도록 설계하는 로직이었다.  
이 부분은 단순 프롬프트 튜닝보다 **시스템 신뢰성을 확보**하는 게 더 중요했다.

### 8-3. JSON schema와 후처리 적용

LLM 프롬프트만으로 전반적인 결과물의 포맷 안정성을 완전히 확보하는 데는 한계가 있었다.  
그래서 agent flow 중간의 출력들은 JSON schema로 형식을 강제했고, Markdown 특수문자나 포맷 깨짐은 Python 후처리로 보정했다.

이 과정을 거치며 얻은 결론은 다음과 같다.
- **LLM은 해석과 서술에 집중시키고, 정합성과 재현성이 필요한 부분은 코드로 통제해야 한다.**
- 이것이 agent간의 Context switching 비용을 늘릴수는 있지만, 결과물의 품질을 훨씬 더 안정적으로 만들어준다.

## 9. RAG는 어떻게 설계했는가

리포트 품질을 끌어올리는 데 중요한 역할을 한 것은 RAG 컨텍스트 설계였다.  

분석 결과에서 실제로 유의미한 인사이트를 얻기 위해서는 **실무진의 도메인 지식**이 필수적이다. 
그러나 우리가 아무리 고객사에 대한 정보를 최대한 반영하려고 노력한다고 해도 실제로 매장을 운영하는 실무진의 도메인 지식을 모두 반영하는 것은 불가능하다. 

가장 간단하게 실무진의 도메인 지식을 반영하는 방법은, LLM 서비스를 이용할때마다 실무진이 도메인 지식을 입력하는 것이다.  
하지만 이는 애초에 "분석을 위한 리소스가 부족하다"는 고객사의 불만을 해결할 수 있는 방안이 아니었다.  

고민끝에 내부 DB에 저장된 고객사의 정적 정보와 현업의 매장 운영 중 축적된 보드 메모 같은 **동적 맥락**을 함께 활용하기로 결정했다.

* **정적 정보**: 주소, 업종, 과거 분석 결과, 기본 매장 정보
* **동적 정보**: 휴무일, 리뉴얼, 프로모션, 현장 특이사항

처음에는 정적 정보만으로도 충분할 것이라고 생각했다. 하지만 실제로는 운영 메모가 들어갔을 때 해석의 질이 확연히 달라졌다. 예를 들어 방문객이 30% 증가했다는 숫자만으로는 변화의 이유를 설명하기 어렵지만, “리뉴얼 오픈 이벤트 진행”이라는 운영 메모가 함께 들어가면 증가 원인을 더 자연스럽고 설득력 있게 해석할 수 있었다.

이 프로젝트에서 RAG의 가치는 단순히 배경지식을 넣는 것이 아니라, **숫자에 맥락을 붙이는 데 있었다.**

## 10. 결과와 성과

이 프로젝트의 성과는 자동화된 PDF 리포트 몇 개가 아니라,  
mAsh 안에서 사용자가 질문을 입력하면 외부 조사, 데이터 분석, 전략 제안까지 이어지는 **자동 분석 리포트 기능 자체**였다.

예를 들어 사용자가 “이번 달 방문객이 줄어들었는데, 상권 영향인지 매장 영향인지 알고 싶다”라고 입력하면, 시스템이 관련 외부 자료를 조사하고 내부 행동 데이터와 결합해 현황 리포트와 전략 리포트를 생성하도록 만들었다. 

### 주요 성과

* 수작업 기준 2~3주 걸리던 상권 분석 리포트 제작 구조를 자동화
* 질문 입력부터 완성 리포트 생성까지 10~15분 내 처리
* 12개 Stage 프롬프트 및 8단계 Agent Flow 설계
* 멀티 LLM 역할 분리 및 적용
* AWS Bedrock KB 기반 RAG 파이프라인 구성
* 아모레퍼시픽, 후지필름 등 다수 고객사에 기능 제공
* 고객 미팅에서 시연 후 계약 연장 및 추가 도입 논의에 활용

여기서 중요한 성과는 단순 자동 생성이 아니라, **해석형 분석을 고객이 직접 사용할 수 있는 기능으로 바꿨다는 점**이다.

## 11. 이 프로젝트의 의미

이 프로젝트는 두 가지 측면에서 의미가 있었다.

첫째, 데이터 분석가로서의 역할이 **단순 리포트 작성에 머물지 않았다**는 점이다.  
나는 고객 질문을 분석 가능한 구조로 재정의하고, 그 과정을 반복 가능한 시스템으로 바꾸는 역할을 했다. 이 점에서 이 프로젝트는 전통적인 분석 프로젝트보다 한 단계 더 제품화에 가까웠다.

둘째, **LLM을 도입하는 방식에 대한 기준을 세웠다**는 점이다.  
이 프로젝트를 거치며 분명해진 것은, LLM을 많이 쓰는 것보다 어디에 써야 하고 어디에는 쓰지 말아야 하는지를 구분하는 판단이 훨씬 중요하다는 사실이었다. 이 경험은 이후 다른 자동화 프로젝트를 설계할 때도 기준이 됐다.

## 12. 한계와 회고

한계도 있었다.
- 멀티 LLM 구조는 속도 측면에서는 유리했지만, 전문가가 작성하는 리포트에 비해서는 품질이 확실히 낮았다.  
- 또한 서울 외 지역은 생활인구 데이터 커버리지가 부족해 지역 확장성에 제약이 있었다.  
- validator와 가드레일을 구축했음에도 최종 납품 전에는 여전히 사람의 검토가 필요했다.  
- 출력 포맷 역시 간헐적인 예외가 남아 있었다.

다시 한다면 세 가지를 먼저 보완할 것이다.
- 첫째, 단계별 중요도에 따라 모델 티어를 더 세밀하게 나눠 비용을 최적화할 것이다.
- 둘째, 서울 외 지역까지 확장 가능한 대체 데이터 소스를 초기에 확보할 것이다.
- 셋째, 프롬프트와 출력 품질을 정량 비교할 수 있는 평가 파이프라인을 더 빨리 만들 것이다.

이 프로젝트를 회고하자면,  
엔터프라이즈 고객이 필요로 하는 것은 데이터 접근 자체가 아니라, **데이터와 외부 맥락을 결합해 실제 행동으로 이어지는 해석형 분석 서비스**라는 점이다.  
그리고 그 서비스를 만들기 위해서는 분석 역량, 시스템 설계 역량, AI 활용 역량이 함께 필요했다.
