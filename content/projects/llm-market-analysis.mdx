## Context and Problem

이 프로젝트의 출발점은 기술이 아니라 한 통의 미팅 요청이었다.

아모레퍼시픽은 당사의 최고가 플랜인 Tracking Plan을 구독 중이었는데, 대시보드 접속 빈도가 눈에 띄게 낮았다. 내부적으로 이탈 조짐이라는 판단이 나왔고, 미팅을 잡았다. 고객의 이야기는 솔직했다. "데이터 자체는 좋은데, 일상 업무가 많은 현업 입장에서 대시보드를 직접 해석하고 외부 상권 정보까지 결합해서 액션 플랜까지 뽑아내기엔 리소스가 없다."

데이터가 부족한 게 아니었다. **해석이 부족했다.** 방문객이 줄었을 때, 매장 운영 문제인지 상권 자체가 쇠퇴하는 건지에 따라 해야 할 일이 완전히 다르다. 상권은 성장하는데 특정 매장만 떨어지고 있다면? 원인은 매장 자체의 경쟁력일 가능성이 높다. 고객이 원한 건 숫자 조회가 아니라 원인 해석과 실행 가능한 액션이었다.

그런데 상권 정보를 확보하는 일 자체가 만만치 않았다. 외부 시장 조사에 지역 상권 분석, 도메인 트렌드 파악까지 더하면 사실상 전문 리서치 업무다. 한 고객사 한 지점에 수개월을 매달리는 방식으로는 서비스가 성립하지 않는다. 결국 풀어야 할 문제는 명확했다 — **외부 맥락을 포함한 분석 리포트를 반복 가능한 구조로 만드는 것.**

프로젝트는 약 2년에 걸쳐 수작업 리포트 제작(Phase 1)에서 LLM 자동화 연구(Phase 2)를 거쳐, 프로덕션 LLM 자동 리포트 파이프라인(Phase 3)으로 발전했다.

## Approach and Design

### Phase 1: Manual Report Insights

처음에는 모든 걸 손으로 했다. 행복한백화점 리포트 하나를 만드는 데 — 데이터 추출, 외부 상권 조사, 분석, 시각화, 스토리라인 구성, 24페이지 PDF 작성까지 — 체감상 2~3주가 꼬박 들었다. 리서치만 해도 해당 지역 유동인구 통계를 찾고, 경쟁 매장 현황을 파악하고, 최근 상권 이슈를 정리하는 데 며칠이 걸렸다.

그 과정에서 분석 구조가 잡혔다. 2단 구조 분석 프레임워크라고 이름 붙인 건데, (1) 전체 방문객 분석(성별/주중-주말/연령대/방문시간대)과 (2) 각 층별 방문객 분석(체류고객 성별비율-체류시간, 연령대, 시간대별, 입구별 트래픽)으로 나누어 거시적-미시적 관점을 모두 제공하는 방식이다.

방문객 분류 체계도 이때 만들었다:
- 실질 방문객: 매장 내 1개 이상의 구역에 60초 이상 머무른 고객
- 상담 전환: 매장 직원과 20분 이상 동행 + 상담 구역 10분 이상 체류
- 배회객: 어떤 쇼룸에서도 5분 이상 체류하지 않은 고객
- 통행객: 에스컬레이터만 이용하고 매장 쇼룸 미입장

Phase 2에서 ChatGPT에 상권 분석을 시켜보면서 깨달은 게 있다. 그냥 "이 데이터를 분석해줘"라고 던지면 그럴듯하지만 쓸 수 없는 결과가 나온다. 그런데 Phase 1에서 직접 작성했던 리포트의 분석 순서 — 어떤 데이터를 먼저 보고, 어떤 맥락에서 비교하고, 어떤 기준으로 판단하는지 — 를 프롬프트에 구체적으로 녹이면 품질이 확연히 달라졌다. "분석해줘"와 "이 데이터에서 전월 대비 변화율을 구하고, 상권 평균과 비교한 뒤, 차이가 10% 이상이면 원인 가설을 3개 제시하라"는 전혀 다른 결과를 만든다. 이 경험이 Phase 3의 12 Stage 프롬프트 설계로 직접 이어졌다.

### Key Design Decisions

> **Decision 1: 파이프라인 설계를 바로 시작할 것인가, 수작업 리포트를 먼저 만들어볼 것인가?**
>
> **검토한 대안:**
> - 바로 LLM 파이프라인 프로토타입 개발 -- 빠르지만 요구사항이 불명확
> - 기존 템플릿 기반 자동 채우기 -- 한계가 명확
>
> **선택:** 직접 수작업 리포트를 먼저 제작한 뒤 자동화
>
> **근거:** 필요한 데이터, 분석 순서, 결과물 구조를 먼저 정리해야 무엇이 자동화 가능하고 무엇은 사람이 판단해야 하는지 구분할 수 있었다. 이 과정이 12 Stage 프롬프트 설계의 기반이 되었다.

> **Decision 2: 내부 데이터와 외부 데이터를 결합하는 과정을 LLM에 맡길 것인가, 코드로 처리할 것인가?**
>
> **검토한 대안:**
> - LLM이 데이터 결합까지 처리 -- 초기 시도, 출력이 매번 미묘하게 달라짐
> - 전체를 코드로 처리 -- LLM 활용 범위가 너무 좁아짐
>
> **선택:** 기계적 처리는 Python 코드, 해석과 서술은 LLM으로 분리
>
> **근거:** 초기에 LLM으로 시도했을 때, 같은 입력에 대해 합계가 미묘하게 달라지거나 비율 계산이 틀어지는 일이 반복됐다. 파이프라인 전체의 신뢰성을 위해 정합성이 필요한 구간은 코드로 빼고, LLM은 해석과 서술에만 집중시켰다.

> **Decision 3: 단일 LLM 모델 vs 멀티 LLM 모델 전략?**
>
> **검토한 대안:**
> - GPT-4.1 단일 모델 -- 단순하고 비용 예측 쉬움
> - 오픈소스 모델 셀프호스팅 -- 비용은 낮지만 운영 부담
>
> **선택:** 분석 단계별 최적 모델 선택 (o3-mini / GPT-4.1 / sonar-pro)
>
> **근거:** 추론이 필요한 분석에는 o3-mini의 reasoning, 자연어 생성에는 GPT-4.1의 유창성, 웹 검색에는 sonar-pro의 실시간성이 각각 최적. 단일 모델보다 비용은 높지만 전체 리포트 품질이 유의미하게 향상.

### Production Pipeline Architecture

Phase 1에서 손으로 2~3주 걸리던 작업을 10~15분으로 줄이는 게 목표였다. MashChain이라는 DAG 기반 노드 실행 엔진 위에 상권 분석 전용 8단계 워크플로우를 얹었다.

가장 먼저 실행되는 건 두 개의 리서치 노드다. business_domain_research와 commercial_area_research가 병렬로 돌면서, Perplexity sonar-pro를 통해 "이 브랜드가 최근 어떤 전략을 쓰고 있는지", "이 상권의 유동인구 추세와 경쟁 환경이 어떤지"를 실시간으로 수집한다. Phase 1에서 며칠 걸리던 리서치가 여기서 수 분으로 압축된 셈이다.

리서치 결과가 모이면 analysis_plan 노드가 데이터 스키마와 합쳐서 분석 계획을 세운다 — 3-level 목차와 각 섹션별 가이딩 질문을 자동 생성한다. 그 다음 data_analysis 노드가 매장 방문객과 생활인구 DataFrame별로 o3-mini를 돌려 차트 스펙과 내러티브를 만든다. 이 단계는 DataFrame 수만큼 병렬로 실행된다.

분석이 끝나면 performance_report 노드가 결과를 종합해 현황 보고서를 작성하고(o3-mini, reasoning_effort=high), 이어서 strategy_report_plan이 전략 보고서 구조를 잡은 뒤 search_query → trend_research를 거쳐 최신 트렌드와 벤치마크를 추가 검색한다. 마지막으로 strategy_report가 모든 걸 엮어 최종 전략 보고서를 완성한다.

각 단계에 들어가는 프롬프트는 직접 설계했다. 총 761줄의 report_prompts.py에 15개 프롬프트가 담겨 있는데, 설계할 때 계속 돌아온 원칙이 네 가지 있다:

1. 하나의 단계에서 하나의 역할만. "분석하고 요약하라"가 아니라 "이 데이터를 분석하라. 요약은 다음 단계에서 한다"로 분리
2. Hallucination 방지 지시와 통계적 근거 기반 주장 제한을 프롬프트 앞부분에 배치 — 뒤에 넣으면 긴 입력에서 묻히는 경우가 잦았다
3. "Senior retail analyst"처럼 전문적 역할을 부여해서 톤과 분석 깊이를 조절
4. 중간 전달은 JSON/YAML, 최종 리포트는 마크다운 템플릿으로 출력 형식 고정

### Quality Control and Reliability

사실 파이프라인을 처음 돌렸을 때 결과물은 꽤 그럴듯했다. 문제는 두 번째, 세 번째 돌렸을 때 드러났다. **수치가 미묘하게 흔들렸다.**

한번은 LLM이 주중 평균 방문객 수를 집계하면서 특정 요일 데이터를 빠뜨려 실제보다 15% 낮은 숫자를 리포트에 넣은 적이 있다. 더 위험했던 건, 문장 자체는 자연스러워서 읽기만 해서는 틀렸다는 걸 알기 어려웠다는 점이다. 고객사에 그대로 나갔다면? 생각만 해도 아찔했다.

이걸 계기로 구조를 바꿨다. 수치를 집계하는 단계와 그 수치로 리포트를 쓰는 단계를 완전히 분리했다. 하나의 단계에서 계산과 서술을 동시에 시키면 오류가 섞여 들어올 여지가 너무 많았다.

거기에 validator 단계를 추가했다. 집계 결과와 최종 리포트를 함께 넣어서, 리포트에 쓰인 숫자가 실제 집계 값과 일치하는지 교차 검증하도록 만들었다. 초기에는 프롬프트 rule — "반드시 원본 데이터의 수치만 사용하라" 같은 — 만으로 통제하려 했는데, 입력이 길어질수록 규칙이 슬금슬금 무시되더라. 결국 구조적으로 강제하는 수밖에 없었다.

포맷 안정성도 문제였다. JSON 스키마를 사전 정의해서 출력 형식을 강제하고, validator 실패 시에는 실패 사유를 해당 stage로 되돌려 재생성하게 했다. Markdown에서 특수문자 때문에 문서가 깨지는 문제는 — 프롬프트로 아무리 "특수문자 쓰지 마라"고 해봐야 LLM은 가끔 무시한다 — Python 문자열 후처리로 잡았다.

결국 도달한 결론은 단순하다. LLM은 해석과 서술에 집중시키고, 정합성과 재현성이 필요한 부분은 코드로 처리해야 한다. 단순한 원칙인데, 이걸 체감하기까지 꽤 많은 실패 리포트를 봐야 했다.

### RAG Context Design

AWS Bedrock Knowledge Base와 mAsh 보드 텍스트 메모리를 조합해서 각 분석 노드에 매장별 컨텍스트를 주입하는 RAG 파이프라인을 구현했다.

- KB: 정적인 매장 정보 (주소, 업종, 과거 분석 결과)
- 보드 메모리: 동적인 운영 메모 (최근 이벤트, 특이사항)
- 매장별 필터링: place_name 기반으로 관련 컨텍스트만 추출

처음에는 KB만으로 충분할 거라 생각했다. 그런데 실제로 돌려보니 차이가 뚜렷했다. 예를 들어 특정 매장의 방문객이 지난달 급증한 경우, KB만으로는 "방문객이 30% 증가했다"는 서술밖에 나오지 않았다. 보드 메모리에 "3월 리뉴얼 오픈 이벤트 진행"이라는 운영 메모가 있으면 LLM이 "리뉴얼 오픈 이벤트의 효과로 방문객이 30% 증가했으며, 이벤트 종료 후 유지율 추적이 필요하다"는 식으로 **맥락이 붙은 해석**을 만들어냈다. 정적 정보와 동적 운영 맥락의 결합이 리포트 품질을 한 단계 끌어올린 셈이다.

## Impact and Outcome

최종 결과물은 별도 문서 납품이 아니었다. mAsh 안에서 사용자가 질문을 입력하면 상권 조사와 데이터 분석을 거쳐 리포트가 생성되는 기능 — 그 자체가 결과물이었다. "이번 달 방문객이 줄어들었는데 상권의 영향인지 알고 싶다"라고 입력하면, 외부 자료 조사부터 데이터 결합, 분석, 리포트 작성까지 10~15분 안에 완료된다.

아모레퍼시픽 미팅에서 이 기능을 시연했을 때, 처음엔 반응이 조용했다. 질문을 입력하고 실시간으로 리서치가 진행되고 리포트가 만들어지는 과정을 지켜보다가, 완성된 리포트에서 자기네 매장 주변 상권 변화와 경쟁 매장 동향이 구체적으로 서술된 부분을 보고 나서야 분위기가 바뀌었다. "이게 자동으로 되는 거냐"는 질문이 나왔고, 이후 대시보드 접속 빈도가 눈에 띄게 올라갔다. **계약 연장과 추가 지점 도입 의사 확보**로 이어진 건 그 이후의 일이다.

Phase 1 성과:
- 행복한백화점 방문객 분석 리포트 제작 및 납품 (24페이지, 7.2MB)
- LG전자, 스파크펫 등 다수 고객사 리포트 납품
- 인사이트 리포트 공식 샘플 템플릿 확립 (19페이지, 방문객 분류 체계 포함)

Phase 3 성과:
- 12개 stage 프롬프트 설계 및 적용 (report_prompts.py 761줄)
- 8단계 DAG 기반 agent flow 워크플로우 설계 및 구현
- 멀티 LLM 모델 전략 설계 및 적용
- AWS Bedrock KB 기반 RAG 파이프라인 구현
- 질문 입력에서 현황+전략 보고서 자동 생성까지 프로덕션 시스템 완성

이후 이 자동화 플로우는 후지필름, 코자 등 다른 고객사에도 제공되었다.

## Limitations and Lessons Learned

솔직히 이 프로젝트를 한마디로 요약하면, "LLM을 얼마나 많이 쓰느냐"가 아니라 **LLM이 해야 할 일과 하지 말아야 할 일의 경계를 찾는 과정**이었다. 멋진 파이프라인을 만드는 것보다, 어디서 LLM을 빼고 코드를 넣을지 판단하는 게 더 어려웠다.

엔터프라이즈 고객에게 필요한 건 데이터 접근성이 아니다. 데이터와 외부 맥락을 연결해서 실제 액션으로 이어지게 만드는 해석형 분석 서비스 — 이걸 2년에 걸쳐 몸으로 배운 셈이다.

한계:
- 멀티 LLM 호출 비용: 8단계 워크플로우를 전부 돌리면 비용과 시간이 무시할 수 없는 수준이었다
- 서울 외 지역 데이터 커버리지: 서울 열린데이터 광장의 생활인구 데이터는 풍부하지만, 부산 같은 지역은 시간대별 데이터가 아예 없었다
- 최종 품질 검증의 인간 의존: 가드레일과 QA validator를 달아놨지만, 고객사에 나가기 전 마지막 검토는 여전히 사람 몫이었다
- 프롬프트 출력 포맷의 edge case: JSON 스키마를 강제해도 LLM이 포맷을 어기는 경우가 간헐적으로 발생

다시 한다면 — 그리고 이건 꽤 구체적으로 생각해본 건데:
- 중요도가 낮은 단계에 mini/haiku급 모델을 배치하는 비용 티어링을 더 세밀하게 가져갔을 것이다. 모든 단계에 고성능 모델을 쓸 필요는 없었다.
- 서울 외 지역은 처음부터 통신사 유동인구나 카드매출 데이터 같은 대체 소스를 확보해뒀어야 했다. 나중에 지역 확장 요청이 들어왔을 때 데이터가 없어서 발이 묶인 적이 있다.
- 프롬프트 변형별 출력 품질을 체계적으로 비교하는 평가 파이프라인을 먼저 구축했을 것이다. 지금은 사람이 읽어보고 "이게 더 낫다"고 판단하는 방식인데, A/B 테스트 기반으로 정량화할 수 있었다면 프롬프트 개선 속도가 훨씬 빨랐을 것이다.
