## Executive Summary

- *채용 담당자가 13개 프로젝트 정보를 빠르게 탐색하려면 어떤 인터페이스가 필요한가?*
- *외부 벡터 DB 없이도 실용적인 RAG 시스템을 구축할 수 있는가?*
- *비용 효율적인 LLM 조합으로 프로덕션 수준의 챗봇을 만들 수 있는가?*

## 배경 & 문제 상황

포트폴리오 사이트를 만들면서 한 가지 고민이 있었다. 13개 프로젝트가 각각 개별 MDX 페이지로 존재하는데, 채용 담당자가 이걸 처음부터 끝까지 다 읽을까? 현실적으로 어렵다. 대부분은 2~3개만 훑어보고 판단할 텐데, 그러면 나머지 프로젝트에서 보여줄 수 있는 역량이 묻힌다.

예를 들어 "이 사람이 LLM을 실제로 써본 경험이 있나?"라는 질문에 답하려면, LLM 상권 분석 자동화 프로젝트를 찾아서 읽어야 한다. "팀을 리드한 경험이 있나?"면 DA 팀 세팅 프로젝트를. 포트폴리오 전체를 꿰뚫어 보는 인터페이스가 있다면 — 자연어 질문 하나로 관련 정보를 바로 찾을 수 있다면 — 훨씬 효과적일 거라는 판단이었다.

동시에 기술적 제약도 고려해야 했다. 개인 포트폴리오 사이트이므로:
- **비용**: 월 고정 비용 없이 쿼리당 과금만으로 운영해야 함
- **인프라**: Supabase, Pinecone 같은 외부 벡터 DB 없이 구현해야 유지보수 부담이 없음
- **데이터량**: 13개 프로젝트 + 경력 정보는 전체를 메모리에 올려도 무리 없는 규모

이 조건에서 실용적인 RAG 챗봇을 만드는 것이 목표였다.

## Approach and Design

### 전체 아키텍처

시스템은 크게 **빌드 타임**(인덱스 생성)과 **런타임**(검색 + 응답)으로 나뉜다.

**빌드 타임:**
1. 5종 데이터 소스에서 텍스트 수집
2. JSX/HTML 태그 스트리핑
3. 문단 기반 청킹 (~500 토큰)
4. `text-embedding-3-small`로 배치 임베딩
5. JSON 파일로 저장 (`rag-index.json`, ~9MB, 280개 벡터)

**런타임:**
1. 사용자 질문 → `text-embedding-3-small`로 임베딩
2. 280개 청크와 코사인 유사도 계산
3. Top-8 결과 추출 + 프로젝트 슬러그 파싱
4. 시스템 프롬프트 + 검색 컨텍스트 + 대화 히스토리 → `gpt-4o-mini`
5. 응답 텍스트 + 관련 프로젝트 슬러그 반환
6. 프론트엔드에서 마크다운 렌더링 + 프로젝트 링크 표시

### 데이터 소스 설계

RAG 시스템의 품질은 검색 대상 데이터의 품질에 직결된다. 어떤 소스를 포함하느냐가 곧 챗봇이 대답할 수 있는 범위를 결정하는 셈이다.

5종 소스를 선정했다:

1. **MDX 프로젝트 파일** (13개): 프론트매터를 제거한 본문 텍스트. 각 프로젝트의 배경, 접근법, 성과가 담겨 있다.
2. **정규화된 프로젝트 overview** (`normalized/projects/*/overview.md`): 각 프로젝트의 구조화된 개요 문서.
3. **정규화된 역할 기술** (`normalized/projects/*/my_role.md`): 프로젝트별 구체적 역할과 기여도.
4. **경력 기술서** (`normalized/career/base_career_description.md`): 전체 경력 맥락과 회사 소개.
5. **강점/약점 문서** (`normalized/strengths_weaknesses.md`): 자기 평가 및 성장 포인트.

특히 MDX 파일과 normalized 문서를 **모두** 포함시킨 이유가 있다. MDX는 서사적이고 상세하지만 구조가 자유로워 핵심 정보가 분산되어 있고, normalized 문서는 간결하고 구조적이지만 맥락이 부족하다. 두 종류를 함께 인덱싱하면, 질문의 성격에 따라 적절한 소스가 검색되는 효과가 있었다. "LG 프로젝트에서 뭘 했나요?"(역할 중심) vs "LG 프로젝트의 기술적 의사결정은?"(서사 중심) 같은 질문이 각각 다른 소스에서 좋은 답을 찾아왔다.

### 청킹 전략

> **Design Decision: 청킹 단위 선택**
>
> **검토한 대안:**
> - 고정 길이 토큰 분할 -- 구현이 단순하지만 문맥이 중간에서 잘림
> - 섹션(h2) 단위 분할 -- 의미 단위 유지되나 한 섹션이 너무 길 수 있음
>
> **선택:** 문단(paragraph) 기반 분할, ~500 토큰 상한
>
> **근거:** 포트폴리오 텍스트는 문단마다 하나의 논점을 다루는 구조. 문단 경계에서 자르면 의미 단위가 자연스럽게 보존되고, 500 토큰 상한으로 임베딩 품질도 유지된다.

구현에서 가장 까다로웠던 부분은 **JSX 스트리핑**이었다. MDX 파일에는 `<MetricHighlight value="280개" label="벡터 청크" />` 같은 React 컴포넌트 태그가 포함되어 있다. 이걸 그대로 청킹하면 두 가지 문제가 발생한다:

1. **임베딩 오염**: JSX 태그가 텍스트의 의미를 희석시켜 검색 정확도가 떨어진다
2. **LLM 혼란**: 검색 결과에 JSX가 포함되면 gpt-4o-mini가 내용을 코드 스니펫으로 인식하고, 분석적 답변 대신 "코드를 설명하겠습니다" 류의 응답을 생성한다

`stripJsx()` 함수로 `<div>` 블록, PascalCase 셀프클로징 컴포넌트(`<MetricHighlight ... />`), HTML 태그, className 속성을 모두 제거했다. 이 처리만으로 인덱스가 289개에서 280개로 줄었고, 응답 품질이 눈에 띄게 개선됐다.

청킹 로직 자체는 `\n\n` (빈 줄)로 문단을 분리한 뒤, 누적 토큰이 500을 초과하면 새 청크를 시작하는 방식이다. 토큰 추정은 `문자 수 * 0.7` — 한글 1자 ≈ 1.5토큰, 영어 1단어 ≈ 1.3토큰을 감안한 근사치다. 정확한 토크나이저를 돌리면 더 정밀하겠지만, 이 규모에서는 근사치로 충분했다.

### 벡터 검색 설계

> **Design Decision: 벡터 저장소 선택**
>
> **검토한 대안:**
> - Supabase pgvector -- 관리형 서비스, 월 고정 비용
> - Pinecone -- 전용 벡터 DB, 오버스펙
> - SQLite + 확장 -- 로컬이지만 배포 복잡
>
> **선택:** JSON 파일 인메모리 로드 + 코사인 유사도 직접 계산
>
> **근거:** 280개 벡터 × 1536차원 = 약 9MB. 서버 메모리에 올려도 부담 없는 크기이고, 검색 시간도 무시할 수 있는 수준이다. 외부 서비스 의존성을 완전히 제거하면서, 단일 `fs.readFileSync`로 인덱스를 로드할 수 있다. 인덱스가 수천~수만 건으로 커지면 벡터 DB를 도입해야 하지만, 이 규모에서 최적의 선택이었다.

코사인 유사도는 직접 구현했다. 280개 벡터에 대해 루프를 돌려도 수 밀리초 수준이라 최적화가 불필요했다. 검색 결과에서 상위 8개(topK=8)를 추출하는데, 이 숫자는 gpt-4o-mini의 컨텍스트 윈도우(128K)와 비용 사이의 균형점이었다. 8개 청크의 평균 크기가 ~300토큰이므로, 컨텍스트에 ~2,400토큰이 추가된다.

인덱스는 서버 프로세스 내에서 싱글턴으로 캐싱한다. 첫 요청에서 JSON 파일을 읽어 메모리에 올리고, 이후 요청에서는 캐싱된 배열을 재사용한다. Next.js API Route의 특성상 서버가 살아 있는 동안 `_index` 변수가 유지되므로, 파일 I/O는 최초 1회만 발생한다.

### 프롬프트 엔지니어링

시스템 프롬프트 설계에서 가장 큰 교훈은 **프롬프트 언어와 응답 언어를 분리**하는 것이었다.

처음에는 한국어로 시스템 프롬프트를 작성했다. "당신은 포트폴리오 도우미입니다. 아래 컨텍스트를 참고하여 한국어로 답변하세요..." 결과는 참담했다. gpt-4o-mini가 거의 모든 질문에 대해 "해당 정보는 포트폴리오에 포함되어 있지 않습니다"라고 거절했다. 컨텍스트에 관련 정보가 분명히 들어 있는데도.

원인을 파악하는 데 시간이 걸렸다. 한국어 가드레일 문구("관련 없는 질문은 정중히 거절하세요")가 지나치게 적극적으로 작동하는 것이었다. gpt-4o-mini는 대형 모델 대비 instruction following이 약해서, 한국어 거절 지시를 과도하게 확대 해석하는 경향이 있었다.

해결책은 시스템 프롬프트를 **영어**로 전환하되, "Answer in Korean"이라는 지시만 추가하는 것이었다. 영어 프롬프트는 gpt-4o-mini가 가장 안정적으로 따르는 언어이고, 거절 범위도 더 정확하게 지킨다.

최종 시스템 프롬프트의 핵심 설계 원칙:

1. **영어 프롬프트 + 한국어 응답 지시**: 모델의 instruction following 안정성 확보
2. **프로젝트 목록 직접 삽입**: 13개 프로젝트명과 핵심 키워드를 프롬프트에 명시하여, 검색 결과가 부족해도 기본 맥락을 유지
3. **마크다운 포맷 지시**: bold, 리스트, 구체적 수치 사용을 명시하여 가독성 높은 답변 유도
4. **명확한 거절 범위**: "completely unrelated to his career (e.g. cooking recipes)"만 거절하도록 범위를 좁게 설정

### 프론트엔드 UX 설계

챗봇 UI에서 가장 신경 쓴 부분은 두 가지였다.

**마크다운 렌더링**: gpt-4o-mini가 생성하는 응답에는 `**bold**`, 리스트, 코드 블록 등 마크다운 문법이 포함된다. 이를 raw text로 보여주면 가독성이 크게 떨어진다. `react-markdown` 라이브러리로 7개 커스텀 컴포넌트(strong, ul, ol, li, p, code, h3)를 정의하여 Tailwind CSS 스타일을 적용했다.

**관련 프로젝트 자동 링크**: RAG 검색 결과에서 프로젝트 슬러그를 추출하여, 어시스턴트 응답 하단에 관련 프로젝트 링크를 자동으로 표시한다. 소스 파싱은 `r.source.match(/^project:(.+)$/)`로 `project:` 접두사가 붙은 소스만 추출하고, 중복 제거 후 프로젝트 상세 페이지로 연결한다. 이 기능이 있으면 "이 프로젝트에 대해 더 자세히 알고 싶으면" → 원클릭으로 해당 케이스 스터디로 이동할 수 있다.

추가로 페이지 상단에 6개 추천 질문 버튼을 배치했다. 이벤트 위임(event delegation) 패턴으로 `data-suggestion` 속성을 읽어 입력창에 자동 채우기하는 방식이다. 처음 방문한 채용 담당자가 "뭘 물어봐야 하지?"라는 진입 장벽을 낮추는 역할을 한다.

## 디버깅 과정

이 프로젝트에서 가장 많은 시간을 쓴 건 코드 작성이 아니라 디버깅이었다. 세 가지 문제가 순서대로 터졌다.

### 문제 1: Windows 터미널 UTF-8 인코딩

개발 중 curl로 API를 테스트했을 때, 한국어 질문이 전부 깨져서 전달되었다. 서버 로그에 `�ڰǿ�` 같은 문자가 찍혔고, OpenAI 임베딩 결과도 엉뚱한 벡터가 나왔다. 영어 질문은 정상 동작했다.

원인은 Windows Git Bash에서 curl을 사용할 때 UTF-8 인코딩이 제대로 전달되지 않는 환경 문제였다. 코드 자체의 버그가 아니었다. 브라우저에서는 정상 동작한다는 것을 확인하고 넘어갔지만, 이 과정에서 한동안 코드를 의심하며 불필요한 디버깅을 했다.

### 문제 2: JSX 태그의 RAG 청크 오염

MDX 파일에서 `<MetricHighlight value="40페이지" label="분석/전략 제안" />` 같은 JSX 태그가 청크에 그대로 포함되자, gpt-4o-mini가 포트폴리오 내용 대신 "코드 블록을 분석하겠습니다" 류의 응답을 생성했다. 모델이 컨텍스트를 코드 문서로 오인한 것이다.

`stripJsx()` 함수를 빌드 스크립트에 추가하여 4가지 패턴(div 블록, PascalCase 셀프클로징, HTML 태그, className 속성)을 제거하고 인덱스를 재빌드했다. 289 → 280 청크로 감소했고, 응답 품질이 즉시 개선됐다.

### 문제 3: OpenAI 프롬프트 캐싱

시스템 프롬프트를 수정해도 응답이 바뀌지 않는 현상이 있었다. 서버 응답의 `cached_tokens: 2816`을 발견하고, OpenAI가 동일한 시스템 프롬프트를 캐싱하고 있다는 것을 파악했다. Next.js 서버 재시작 + `.next` 캐시 삭제로 해결했지만, 프롬프트 변경 후 즉시 테스트할 때 캐시 무효화를 고려해야 한다는 교훈을 얻었다.

## Impact and Outcome

이 프로젝트의 직접적인 효과는 세 가지다.

첫째, **포트폴리오 탐색 경험의 질적 전환**. 13개 프로젝트를 하나씩 읽는 방식에서, "LLM 경험이 있나요?"라는 한 줄 질문으로 관련 프로젝트 3~4개의 핵심 내용을 요약받는 방식으로 바뀌었다. 관련 프로젝트 링크가 자동으로 제공되므로, 관심 있는 프로젝트만 깊이 읽는 효율적인 탐색이 가능해졌다.

둘째, **기술 역량의 실증적 시연**. RAG, 임베딩, 프롬프트 엔지니어링, LLM 파이프라인 설계 — 이력서에 키워드로 나열하는 것과, 실제로 동작하는 시스템으로 보여주는 것은 설득력이 다르다. 특히 LLM 상권 분석 자동화 프로젝트에서 쌓은 경험(멀티 LLM 전략, 프롬프트 설계, 품질 관리)이 이 작은 프로젝트에서도 일관되게 적용된다는 것을 보여준다.

셋째, **비용 효율적 아키텍처 설계 역량 입증**. 외부 벡터 DB 없이 인메모리 JSON으로, 최저 비용 모델 조합(text-embedding-3-small + gpt-4o-mini)으로, 프로덕션 수준의 RAG 시스템을 구현했다. "제약 조건 안에서 실용적인 해결책을 찾는다"는 것이 이 포트폴리오 전체를 관통하는 메시지인데, 이 프로젝트가 그 메시지의 라이브 데모 역할을 한다.

## Limitations and Lessons Learned

**한계:**
- **인메모리 인덱스의 확장 한계**: 280개 벡터는 문제없지만, 수천 건 이상으로 늘어나면 JSON 파일 로드 시간과 메모리 사용량이 병목이 된다. 그 시점에서 벡터 DB 마이그레이션이 필요하다.
- **대화 맥락 제한**: 전체 대화 히스토리를 매번 LLM에 전달하지만, 긴 대화에서는 토큰 비용이 누적되고 오래된 맥락이 희석된다. 대화 요약이나 슬라이딩 윈도우 같은 전략이 없다.
- **검색 품질의 임베딩 의존**: 코사인 유사도 단일 기준으로 검색하므로, 키워드 매칭이 필요한 정확 검색(예: 특정 기술명)에서 약할 수 있다. 하이브리드 검색(벡터 + BM25)을 도입하면 개선 가능하다.
- **실시간 데이터 반영 불가**: 인덱스는 빌드 타임에 생성되므로, MDX 파일을 수정해도 인덱스를 재빌드하지 않으면 반영되지 않는다.

**다시 한다면:**
- 스트리밍 응답(Server-Sent Events)을 구현했을 것이다. 현재는 전체 응답이 완성된 뒤 한 번에 표시되는데, 토큰 단위 스트리밍이 되면 체감 응답 시간이 크게 줄어든다.
- 검색 결과의 유사도 점수를 기반으로 "확신 없음" 표시를 추가했을 것이다. 모든 청크의 유사도가 낮을 때(예: 0.3 이하) LLM에 컨텍스트 없이 질문을 넘기면 hallucination 위험이 높아지는데, 이를 사전에 차단하는 로직이 있으면 더 안전하다.
- 사용자 질문 로그를 수집해서, 자주 묻는 질문과 답변 품질을 분석하는 피드백 루프를 만들었을 것이다. 어떤 질문에서 RAG가 잘 작동하고 어디서 실패하는지를 정량적으로 파악해야 지속적 개선이 가능하다.
