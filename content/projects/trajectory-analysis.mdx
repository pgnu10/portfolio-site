## Trajectory Analysis Module 설계 및 제품화

### 프로젝트 개요

<img src="/images/flow_animation.gif" alt="Trajectory Analysis Module 데모" style={{ width: "100%", borderRadius: "8px" }} />

기존 히트맵만으로는 방문객이 어디에 머물렀는지는 보여줄 수 있었지만, 어떤 순서로 이동했고 어떤 흐름 패턴을 보였는지는 설명하기 어려웠다.
좌표 로그를 시퀀스 단위로 재구성해 유사 동선을 분류하는 FlowMap 프로토타입을 설계하고, 고객사별 매장 구조 차이를 YAML config로 일반화한 뒤 heatmap / journeymap / traffic_map으로 제품 모듈화했다.
동선 데이터를 밀도 중심 분석에서 순서/유형/흐름 분석으로 확장했고, 실제 고객사 데이터에 적용 가능한 재사용형 분석 파이프라인과 프로덕션 모듈 3종을 구축했다.

참고: [메이아이 블로그: 매장 유동인구 데이터, 이렇게 봐야 잘 ‘보입니다’](https://blog.may-i.io/product-mashboard-4/)

## 접근 방식

### 1) 좌표 로그를 집계가 아니라 시퀀스로 재해석

히트맵은 많이 머문 위치는 잘 보여주지만 방문객이 어떤 순서로 공간을 경험했는지는 설명하지 못했다.
- 좌표 로그를 단순 점 집합으로 보지 않고 시간 순서가 있는 trajectory로 재구성했다.
- 이후 경로 유사도를 비교할 수 있는 방식이 필요하다고 판단해 DTW, LCSS, NMF 등을 검토했고, 
- 실제 이동 순서를 보존하면서 비교 가능한 시퀀스 분석 구조를 탐색했다.

동선 데이터는 체류 밀도 분석을 넘어 방문 순서와 이동 패턴을 해석할 수 있는 행동 데이터로 확장됐다.

### 2) 고객사별 공간 차이를 config 기반으로 일반화

신세계, 현대이노션처럼 고객사마다 구역 구성과 좌표 구조가 달라 코드 수정 중심으로는 확장성이 떨어졌다.
- 매장별 구역 정의, 좌표 범위, 시각화 기준, 라벨 정보를 코드가 아니라 YAML config로 분리했다.
- 이를 통해 분석 로직은 공통으로 유지하고, 고객사별 차이는 설정값만 바꿔 대응할 수 있도록 구조를 설계했다.

동선 분석 모듈은 특정 고객사 전용 코드가 아니라 여러 공간 구조에 적용 가능한 재사용형 파이프라인으로 정리됐다.

### 3) 프로토타입을 질문 중심의 3개 모듈로 분화

초기 FlowMap 하나로는 다양한 동선 질문을 모두 담으려다 보니 결과 해석과 활용 방식이 혼재되기 쉬웠다.
- 고객이 실제로 묻는 질문을 기준으로 기능을 나눴다.
- `heatmap`은 “어디에 머무르는가”, `journeymap`은 “어떤 순서로 움직이는가”, `traffic_map`은 “어디에서 어디로 이동하는가”에 답하도록 역할을 분리했다.
실험용 도구 하나를 유지하는 대신, 고객 질문에 맞춰 반복 활용 가능한 분석 모듈 체계를 만들 수 있었다.

## 성과: 좌표 로그 기반 동선 분석을 행동 시퀀스와 흐름 분석으로 확장하고, 이를 재사용 가능한 모듈로 제품화
* trajectory를 시퀀스 단위로 해석하는 분석 프레임 수립
* KMedoids 기반 동선 유형화와 fastDTW 기반 거리 계산으로 R&D 검증
* YAML config 기반 일반화로 고객사별 공간 차이 대응
* heatmap / journeymap / traffic_map 3종 모듈 프로덕션화
* Sankey diagram을 통한 흐름 전달력 강화
* 신세계, 현대이노션 등 복수 고객사 데이터 적용 가능 구조 확보
* 실험성 높은 분석을 실제 고객 프로젝트에서 재사용 가능한 자산으로 정리
